
# CYFS Protocol 0.9 文档 Review



## 结构性问题


### 2. 占位章节应该显式标记 STATUS

当前文档里有这些半成品：
- "透明加速"（明确标了"占位，未做详细设计"）✓
- "内容消费证明"（标了"本章节是协议占位"）✓
- "使用 container_id/@/key 可信地获取对象"（标了"实验性设计，尚未定稿"）✓
- 但 "Zone 间传播" 事实上只给了一个 5 行伪代码，**没标 status**
- "基于 ReferencePath 的权限控制" 只有 3 个 bullet 的概念描述，**也没标 status**

**建议**：在文档头建立统一的 status 标记规范，例如 `STATUS: stable | draft | experimental | placeholder`，并在每一节的开头统一打这个标签。目前每一节 status 的写法不一致（有的是引用块、有的是括号、有的没有），读者不容易一眼定位"哪些是可以实现的、哪些不能动手"。

### 3. "标准对象参考"指向的附录文档本身就是一个矛盾源

文档末尾指向了 `CYFS协议现有标准对象实例.md`。但附录文档第 4 节明确写着：

> "以下类型码已经在 `src/ndn-lib/src/lib.rs` 中保留，但当前仓库里没有稳定、可直接举例的独立结构定义"

这意味着协议的"标准对象列表"**和实现仓库的状态强耦合**。对于一份 0.9 版的协议草案来说，这是个危险信号：
- 协议应该先定义对象，然后才有实现
- 现在是"仓库里实现了什么，协议就说自己有什么"

**建议**：
- 主协议文档里给出一张**权威的标准对象表**（包括那些还没实现的 `cypack`、`cytrie` 等），每个对象标注 `spec_status` 和 `impl_status` 两个独立字段
- 附录文档改名为 `CYFS_StandardObjects_Implementation_Status.md`，只负责追踪实现进度

## 三、技术/规范严谨性问题

这部分是比较具体的 spec 级别 bug 或含糊点。

### 1. 稳定 JSON 编码规则没有完整定义

按建议，现在遵循RFC8785

### 2. ObjectId 的两种编码未定义等价关系

第 27-33 行说 `sha256:<hex>` 和 `base32_rfc4648_encode("sha256:" + hash_data)` 是"等价的"，但没有说：
- base32 编码是否包含 padding？RFC 4648 允许 padding 也允许省略
- 是否区分大小写？base32 标准是大写，但很多实现默认小写
- 在 URL 第三种形式 `http://{base32_objid}.$zoneid/` 里，hostname 是大小写不敏感的，这和 base32 编码的可逆性如何协调？

### 3. `mix256` 是什么？文档里只用不定义

全文多处出现 `mix256:80c00940...`，但**完全没有说明 `mix256` 是什么哈希算法**。从上下文推测是某种混合哈希（可能是 sha256 + 大小前缀？），但这是协议级概念，必须有定义。

附录文档也一样，在 `cyfile` 的 content 字段里直接用，没有解释。读者（和将来的实现者）完全不知道该怎么算。

### 4. `cyfs-path-obj` JWT 的签名算法、Key ID 没指定


建议 `cypath` 统一改成 `iat`，向现有的标准 JWT 生态对齐。

### 5. `inner_path` 语义有一个未处理的边界情况

第 6 条规则："如果中间某一层返回的不是 ObjectId，但 URL 里后面还有新的 `/@/` 段，则该请求非法。"

但没说第 5 条"结果不是 ObjectId 就是最终返回值"里的"值"能不能是一个**嵌套 JSON 对象**。比如 `cymsg.content.machine.data` 本身是个 `{"level":3,"urgent":true}`——用户能不能通过 `/@/content/@/machine/@/data` 一层层走进去？

从 JSON Path 语义上说应该可以，但规则第 1 条说 "inner_path 语义与 JSON Path 的按字段逐层取值一致"，与规则第 6 条存在冲突。需要澄清：**inner_path 是只在 ObjectId 边界断裂，还是会在任何"非 ObjectId 值"断裂？**

### 6. `resp=raw` 的兼容性没说清

第 289-299 行说 `resp=raw` "不需要任何额外 Header，也不需要服务端自动展开更多引用"。但：
- 服务器必须支持它吗？还是可以忽略？ 必须
- 如果对 Chunk URL 加 `resp=raw` 会怎样？ 不附加验证header,但还是返回字节流
- 它和 `inner_path` 组合使用时的语义是什么？ 不展开,inner_path指向的是value就返回value,指向的是objid就是返回objid

### 7. Chunk 32MB 上限的合理性和可扩展性

第 305 行"底层存储系统保存的标准 Chunk 大小上限为 32MB"——这是个硬编码数字。
- 为什么是 32MB 而不是 16MB 或 64MB？（应该给出设计 rationale） ： 从NamedStore底层的块存储角度设计
- 未来能不能提升？如果能，`ChunkId` 格式要不要预留版本字段？ ： 能提升，协议并无限制。只是通过这个方法让所有人构造的chunklist一致


### 8. `SimpleChunkList` 的 ObjectId 算法描述过于含糊

第 322-326 行：
> "`SimpleChunkList` 的 `ObjectId` 不是普通 `NamedObject` 的'稳定 JSON 直接 Hash'，而是由下面两部分共同决定：ChunkId 列表的顺序内容 + 整个文件的总大小 `total_size`。"

附录里也只写了"先对数组内容求 clist 基础哈希，再把总长度编码到 ObjId 中"——**具体算法没给**。这是一个核心标准对象，spec 里必须有精确到字节级的伪代码或公式。

## 四、语言与可读性

整体中文表达清楚，以下是一些具体的小问题：

**第 45 行**："第三种形式在协议上支持，但一般不会默认开启，多用于一些特别大的DirObject。" —— 为什么大 DirObject 需要子域名形式？没有说明。

**第 47 行**："但只要浏览器完整支持了 `cyfs://`，即使使用 `http` 也是相对安全的" —— 这句话容易被误读成 "http 就是安全的"。建议改成"即使底层走 http，传输层虽未加密但内容完整性有 Hash 保障"。

**第 425 行**："**No Push! `cyfs://` 是 `pull-first` 的协议。**" —— 这一句非常有力，但紧接着第 427 行就说"但在很多产品逻辑里，仍然会出现传统的上传行为"，然后讨论了大量上传细节。建议把小节标题改成"内容的发布与 Zone 内上传"，避免前后口径的落差感。

**第 484 行**："这里描述的是 `cyfs://` 里的**君子协定**" —— 这个词很传神，但后面并没有展开"君子协定"到底指什么（是弱强制？还是基于声誉？）。可以一句话解释清楚。

**第 529 行** "Consumer" 首次出现 —— 前面（524行）定义"3 个核心角色"（Owner/Curator/Sharer），然后第 566 行又列出 4 个角色（多了 Consumer）。**到底是 3 个还是 4 个？** 如果是 4 个，上面那段需要改。

先把你回答的三个问题的含义梳理清楚：

**DID 兼容 W3C**——那 spec 里只需要一句话锚定 W3C DID Core，具体 verification method 的解析都走 W3C 推荐路径，不需要 CYFS 自己发明。这反而让我之前担心的 `cyfs-path-obj` JWT 签名算法问题有了明确归属：`alg` 和 `kid` 的规则直接引用 W3C DID + JOSE 即可，CYFS 只负责约束"**哪些 JOSE 算法是 MUST / SHOULD / MAY**"这一层。
**和 IPFS 没关系，源头是迅雷 P2P**——这个背景对文档很有帮助。IPFS/libp2p 走的是"DHT + CID + Bitswap"那一套学院派路线；迅雷 P2P 的基因是"Tracker + 多源调度 + 实际可用的分块下载"，核心是**工程上真的能跑起来的多源加速**。这解释了为什么 CYFS 里"多源发现"、"主动加速"、"Reference 传播链路"这些章节写得这么自然——它们不是从 NDN 论文倒推的，而是从一个真实跑过万亿级流量的 P2P 系统里沉淀下来的。**这个 lineage 应该在文档里明确交代**，原因下面展开。

---

基于这三点，我调整一下 review 的侧重，给你几条针对"准备定稿"这个新目标的建议：

## 一、建议在文档开头加一段"设计谱系与非目标"

现在文档开头只有一句"0.9，最新草案，领先所有实现"。对于一份要长期稳定的协议，读者——尤其是五年后才接触它的读者——需要知道**这个协议是从哪条技术脉络长出来的**，以及**它刻意选择了不做什么**。

建议加的内容大致是：

- CYFS 的内容寻址与多源传输设计，源自 P2P 领域对"大规模多源可验证分发"的长期工程实践，而不是 IPFS/libp2p 谱系。
- CYFS 的身份与可信发布机制，基于 W3C DID Core 扩展，不自定义身份体系。
- CYFS 是 pull-first 的 Content Network 协议，**不是**通用 DHT、**不是**区块链、**不是**密钥分发系统、**不是**访问控制系统——这些能力由上层协议各自解决。

这段"非目标"清单尤其重要。看过太多协议文档因为没写清楚 non-goals，被后来者不断追问"为什么不支持 X"。CYFS 显然做过很多刻意的取舍（比如 No Push），把这些取舍在开头集中说清楚，比散落在各章更容易维护。

## 二、关于"以后肯定不改了"——冻结前的清单

我建议在正式冻结前，把下面这些**会产生永久制品**的决策逐条过一遍，每一条都要有明确的、可以被第二个实现者独立复现的 spec：

| 冻结项 | 当前文档状态 | 冻结前必须补的内容 |
|---|---|---|
| Canonical JSON 编码 | 只有一段 TS 参考实现 | 成文规范：字段排序、Unicode 归一化（NFC）、数字表示、重复 key 处理、禁止值（NaN/Infinity/undefined） |
| `mix256` 哈希算法 | 仅在示例中使用，无定义 | 明确算法：输入、分块策略、与 `sha256` 的关系、为什么需要它 |
| `SimpleChunkList` 的 ObjectId | 文字描述"基础哈希+总长度编码" | 精确到字节的伪代码 |
| ObjectId 的 base32 形式 | 两种等价未明确 | padding、大小写、在 hostname 中的归一化规则 |
| 32MB Chunk 上限 | 硬编码数字 | 是否允许未来扩展？如果允许，`ChunkId` 是否需要版本前缀？ |
| `cypath` 字段命名（`uptime` vs `iat`） | 当前是 `uptime` | 这是冻结前最后的机会改成 `iat` 与其他对象对齐 |
| 标准对象类型码列表 | 附录里列了一批"保留但未实现"的 type code | 要么确认保留（写进主 spec），要么现在删掉；冻结后再加很难 |

这里面最紧迫的是**Canonical JSON**。这一条如果不定死，未来任何一个语言的实现（Go、Python、Swift）都可能因为 JSON 库的微小差异产生不同的 ObjectId，整个协议的可验证性就崩了。我强烈建议直接基于 **RFC 8785 (JCS)** 写，或者明确声明"CYFS Canonical JSON 是 RFC 8785 的子集，额外约束如下..."——站在已有 RFC 上比自己定一套稳得多。


## 四、六年的历史本身是资产

这份文档写下第一行到现在已经六年了——这个事实本身值得在文档里留个痕迹。不是为了怀旧，而是因为：

1. 六年意味着这些设计不是拍脑袋的，是经过时间检验的。
2. 新读者看到"2019 起草、202X 定稿"会比看到"0.9 草案"更愿意认真读。
3. 对于你自己和 BuckyOS 团队来说，这是一份从迅雷 P2P → NDN 学界分享 → CYFS → BuckyOS 的技术谱系的延续，值得被明确记录。

建议在文档开头加一段简短的 history note，类似："本协议源自 2019 年的早期设计，吸收了 P2P 分发领域的工程经验和 NDN 学术社区的讨论，于 202X 年定型"。这种背景信息不影响 spec 的严谨度，但会让读者对文档产生信任。
